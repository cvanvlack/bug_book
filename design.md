Bug Book PWA — Developer Spec (v1)

1. Goal

Build a very lightweight personal progressive web app for recording daily “Bug Book” journal entries.

The app should be:

* fast to open
* optimized for quick entry
* installable as a PWA
* hosted as a static site on GitHub Pages
* backed by a Google Sheet via a minimal Google Apps Script endpoint

GitHub Pages is a static hosting service for HTML/CSS/JS, so the write path needs to go to a separate service endpoint rather than relying on server-side code inside Pages itself. Google Apps Script supports doPost(e) web apps and can return raw text/JSON via ContentService, which fits this architecture well.  ￼

⸻

2. Product scope

In scope

A single-screen entry form that submits one journal row at a time to Google Sheets.

Out of scope

* multi-user support
* authentication/login
* editing entries from the app
* deleting entries from the app
* recent history view
* analytics/dashboard inside the app
* offline queueing or sync
* duplicate-date prevention
* rigorous validation beyond basic field correctness

This is intentionally “cheap and cheerful.”

⸻

3. Users

Single user only: you.

There is no formal auth. The system will use a shared secret/API key approach saved in browser-local storage for the PWA and also checked by the Apps Script endpoint. This is not strong security; it is just possession-based protection.

⸻

4. High-level architecture

Frontend

Static PWA hosted on GitHub Pages.

Recommended stack for v1:

* plain HTML/CSS/JavaScript, or
* a very small build setup such as Vite if the developer prefers

Given the simplicity, plain JS is probably enough.

Backend

Google Apps Script web app with:

* doPost(e) endpoint
* validation of shared secret
* append of submitted row into a Google Sheet

Data store

One Google Sheet tab for entries.

This architecture is compatible with GitHub Pages static hosting and with Apps Script web apps.  ￼

⸻

5. Core UX

Main screen

Single fast-entry form only.

No history panel. No charts. No recent entries.

Submission flow

1. User opens app
2. Date defaults to today in local timezone
3. User fills fields
4. User taps submit
5. App sends entry to Apps Script
6. On success:
    * show success message
    * include the entry date in the success message
    * clear the form for the next entry
    * reset date default back to today
7. On failure:
    * show clear error message
    * keep entered values intact so they can retry

⸻

6. Fields

Each submission must contain these fields:

6.1 Entry date

* visible/editable field
* default: today in local timezone
* user can change it to backfill prior days
* stored as the intended entry date, not just submission day

6.2 Score

Allowed values:

* +2
* +1
* 0
* -1
* -2

This should be a required field.

Preferred UI:

* segmented control / pill buttons / radio group
* not free text

6.3 Creative minutes

* dropdown field
* required
* 15 minute increments
* minimum 0
* maximum 12 hours

Examples:

* 0 minutes
* 1 hour
* 2 hours 30 minutes

6.4 Social minutes

* dropdown field
* required
* 15 minute increments
* minimum 0
* maximum 12 hours

6.5 Brief description of the day

* short text input or textarea
* required

6.6 Why it got this score

* textarea
* required

6.7 Submission timestamp

Not user-entered, but stored automatically by the system.

You asked for local date/time storage. Because browser local time and Apps Script server time can differ, the cleanest approach is:

* frontend sends:
    * entry date chosen by user
    * client-generated local submission timestamp in ISO form
    * client timezone identifier if available, e.g. America/Toronto
* backend also records:
    * server receipt timestamp

That gives you both the user-intended local time and a reliable server receipt time.

⸻

7. Data model

Each row in Google Sheets should contain:

1. entry_date
    * the date the entry is about
    * example: 2026-04-17
2. score
    * one of 2, 1, 0, -1, -2 or string forms +2, +1, 0, -1, -2
    * pick one canonical format and keep it consistent
    * recommendation: store as numeric values 2, 1, 0, -1, -2
3. creative_minutes
    * integer number of minutes
4. social_minutes
    * integer number of minutes
5. day_description
    * text
6. score_reason
    * text
7. submitted_at_local
    * ISO timestamp from browser
    * example: 2026-04-18T09:14:52-04:00
8. client_timezone
    * example: America/Toronto
9. received_at_script
    * timestamp generated by Apps Script when row is appended
10. api_version

* example: v1

Optional but useful:
11. user_agent
12. source

* example: pwa

⸻

8. Google Sheet structure

Use one spreadsheet with one primary tab, for example:

Tab name: entries

Header row:

entry_date | score | creative_minutes | social_minutes | day_description | score_reason | submitted_at_local | client_timezone | received_at_script | api_version | user_agent | source

Behavior:

* always append new row
* never update existing rows
* never deduplicate dates
* no enforcement of one-entry-per-day

This matches your journal-style intent.

⸻

9. Validation rules

Frontend validation

Before submit:

* entry_date required and valid
* score required and must be one of allowed values
* creative_minutes required, numeric, >= 0
* social_minutes required, numeric, >= 0
* day_description required, non-empty after trim
* score_reason required, non-empty after trim

Backend validation

Apps Script must re-validate everything and never trust the client.

If invalid:

* return HTTP-style success/fail envelope in JSON
* include human-readable error message

Example invalid cases:

* missing secret
* wrong secret
* score outside allowed values
* invalid date
* non-numeric hours
* negative hours
* blank description/reason

⸻

10. Security model

Chosen model

No login. Possession-based shared secret only.

Implementation

* app includes a configured API secret
* app sends secret with each POST
* Apps Script compares against stored expected secret
* if mismatch: reject request

Important caveat

This is weak protection because anything embedded in a static frontend can be extracted by someone motivated. GitHub Pages serves static client assets publicly, so secrets in the shipped frontend are not truly secret. This is acceptable only because your use case is personal and low-risk. GitHub Pages is for static files, not secure server-side secret handling.  ￼

Recommendation:

* keep the sheet non-public
* accept that this is convenience, not security

⸻

11. Networking behavior

Connectivity

App only needs to work online.

No offline queueing is required.

Retry behavior

For v1:

* no automatic retry queue
* just show error and let user resubmit manually

Timeout

Set a reasonable client timeout, e.g. 10–15 seconds.

⸻

12. PWA requirements

The app should be installable as a PWA.

Minimum requirements:

* manifest file linked from HTML
* service worker registered
* app served over HTTPS
* icons for install
* standalone display mode

MDN’s installability guidance centers on having a web app manifest, and service workers are used to control caching and offline behavior. GitHub Pages serves over HTTPS, which is compatible with PWA hosting.  ￼

PWA behavior for this project

Even though the app does not support offline submission, the service worker should still cache static assets so the shell loads quickly. If the user is offline and presses submit, the app should clearly say that submission requires internet.

Manifest

Include at minimum:

* name
* short_name
* start_url
* display: "standalone"
* background_color
* theme_color
* icons, including maskable if convenient

Service worker

Cache:

* HTML
* CSS
* JS
* icons
* manifest

Do not attempt offline POST replay in v1.

Layout

Single-column mobile-first form.

Controls

* Date picker for entry date

Suggested labels

Button states

14. API contract

Request

POST to Apps Script web app URL

Content type:
application/json

Example request body:

{
  "apiKey": "YOUR_SHARED_SECRET",
  "entryDate": "2026-04-17",
  "score": -1,
  "creativeMinutes": 90,
  "socialMinutes": 30,
  "dayDescription": "Low-energy day, but made some progress.",
  "scoreReason": "Tired and scattered, but still moved things forward.",
  "submittedAtLocal": "2026-04-18T09:14:52-04:00",
  "clientTimezone": "America/Toronto",
  "apiVersion": "v1",
  "userAgent": "optional browser UA string",
  "source": "pwa"
}

Success response

{
  "ok": true,
  "message": "Saved entry",
  "entryDate": "2026-04-17"
}

Error response

{
  "ok": false,
  "message": "Invalid score"
}

Apps Script web apps support doPost(e), and ContentService is the standard way to return raw text or JSON-like text responses from such endpoints.  ￼

⸻

15. Apps Script responsibilities

The Apps Script backend should:

1. receive POST request
2. parse JSON body
3. validate shared secret
4. validate fields
5. open target spreadsheet/tab
6. append row
7. return JSON response

Suggested config values in script:

* spreadsheet ID
* sheet/tab name
* expected shared secret

Deployment

Deploy as a web app with access configured so the PWA can call it directly. Apps Script web apps are specifically intended for browser-accessible endpoints via doGet / doPost.  ￼

⸻

16. GitHub Pages responsibilities

The GitHub Pages site should:

* host the static frontend
* include manifest and icons
* register service worker
* provide a settings flow that stores the following locally in the browser:
    * Apps Script endpoint URL
    * shared secret
* keep non-secret app defaults in code (for example app name/version)

GitHub Pages publishes static files pushed from a repository or from a Pages workflow.  ￼

⸻

17. Error handling requirements

Must handle:

* no internet connection
* Apps Script endpoint unavailable
* invalid secret
* spreadsheet append failure
* malformed payload
* user submits twice quickly

Required behaviors:

* never silently fail
* always show visible status
* keep form contents on failure
* clear only on success

⸻

18. Non-functional requirements

Performance

* page should load quickly on mobile
* submission should feel immediate
* minimal dependencies preferred

Reliability

* append-only flow should be resilient
* backend should log errors for debugging

Apps Script provides logging facilities, though they are mainly for development/debugging and not long-term observability.  ￼

Maintainability

* keep code small and understandable
* avoid unnecessary framework complexity
* keep all field names aligned between frontend, API, and sheet headers

⸻

19. Nice-to-have but not required for v1

Do not implement unless there is time:

* dark mode polish
* install prompt UX
* keyboard shortcuts
* last-used values remembered locally
* export/import
* local draft autosave
* CSV backup
* trend view
* duplicate warning for same date

⸻

20. Acceptance criteria

The app is done when:

1. It is reachable from GitHub Pages.
2. It can be installed as a PWA on supported devices/browsers with manifest + service worker in place.  ￼
3. Opening the app shows a single fast-entry form.
4. Date defaults to today in local timezone.
5. User can change date to backfill prior days.
6. Score can only be selected from +2, +1, 0, -1, -2.
7. Creative and social hours accept non-negative numeric values.
8. Description and score-reason are required.
9. Submitting while online appends exactly one new row to Google Sheets.
10. The row contains both the chosen entry date and a recorded timestamp.
11. No duplicate-date prevention occurs.
12. No edit/delete functionality exists in the app.
13. On success, the form clears and a success message includes the saved date.
14. On failure, the form remains filled and an error message is shown.

⸻

21. Recommended implementation approach

For the developer, I would recommend this sequence:

Phase 1

Build static form locally with validation only.

Phase 2

Create Apps Script endpoint and test POST manually.

Phase 3

Wire frontend submit to endpoint.

Phase 4

Add manifest, icons, service worker.

Phase 5

Deploy frontend to GitHub Pages and backend as Apps Script web app.

Phase 6

Test from desktop and phone.

23. One-paragraph handoff summary

Build a single-user, append-only Bug Book PWA hosted on GitHub Pages, with a single fast-entry form for date, score, creative hours, social hours, brief description, and why the day got that score. The app should submit online-only to a Google Apps Script doPost endpoint that appends rows into a Google Sheet. No login, no edit/delete, no history view, and no duplicate-date enforcement. On success, clear the form and show a confirmation including the entry date; on failure, preserve the entered data and show an error. The app should be installable as a PWA using a manifest and service worker.  ￼
